🏄‍♂️ About Northern Beaches Surf Check

← Back to Surf Report

🌊 Overview

Northern Beaches Surf Check is an automated surf reporting system for Sydney's Northern Beaches. It runs daily at 5:45 AM via a macOS launch agent, fetches live marine and meteorological data, processes it into a detailed surf report, and deploys it as a static webpage via GitHub Pages.

This page explains how the surf report is generated — the data sources, calculations, and decisions behind every rating and recommendation. The system now evaluates three swell components (primary, secondary, tertiary) from the spectral wave model, classifies them by period into Groundswell and Windswell, and uses the best component for each beach's rating.

📡 Data Sources

IMOS Satellite SST (Real-Time Water Temperature)

Instead of relying on monthly averages, the system fetches real-time sea surface temperature from the Integrated Marine Observing System (IMOS) via their public S3 bucket:

  • Product: RAMSSA L4 — Regional Australian Multi-Sensor SST Analysis (9 km resolution, gap-free)
  • Bucket: s3://imos-data/IMOS/SRS/SST/ghrsst/L4/RAMSSA/
  • Access: Public bucket, accessed via aws s3 cp --no-sign-request
  • Query: Downloads the latest NetCDF file, extracts SST for the Northern Beaches coastal edge (first ocean pixel offshore)
  • Coverage: Daily 12:00 UTC analyses, typically available by ~17:00 UTC the same day

If the S3 endpoint is unreachable or the NetCDF file cannot be parsed, the system falls back to monthly climatological averages.

Open-Meteo Marine API

Provides hourly wave forecasts for the Sydney coast (-33.78, 151.30) using the ECMWF WAM spectral wave model:

  • Swell Components — The model partitions the wave spectrum into up to three swell components (primary, secondary, tertiary). Each has its own height, period, and direction. The system evaluates all three and picks the best one per beach.
  • Wave Height — Significant wave height in metres (Hs)
  • Wave Period — Peak wave period in seconds
  • Wave Direction — Direction waves are coming from, in degrees from north
  • Wind Sea — Locally-generated wind chop, shown separately in the conditions table
  • Wind Speed & Direction — 10m wind speed and direction

Tide Data — Harmonic Tide Model (8 Constituents)

The system uses a harmonic tide model with 8 constituents (M2, S2, N2, K2, K1, O1, P1, Q1) calibrated for Fort Denison. See the Tide section below for full details. The model covers >95% of tidal variance.

🌊 Effective Wave Height at Each Beach

The system computes effective height — the face height of the breaking wave at each beach — by applying two corrections to the offshore swell:

1. Headland Diffraction — Wiegel's Curves

Each beach has a primary aspect and an L/R window defined in beaches.json. Swell arriving inside the window is fully exposed (Kd = 1.0). Outside the window, energy is reduced by Wiegel diffraction:

Shadow angleKdHeight retained
0° (inside window)1.00100%
10° past0.7070%
30° past0.4040%
60° past0.22522.5%
90° past0.1515%

The Kd values are further adjusted for wave period: longer-period groundswell (16s) wraps around headlands more effectively, increasing Kd by up to 1.24×, while short-period windswell (6s) diffracts less, reducing Kd to 0.84× the table value.

2. Shoaling Amplification

Longer-period waves carry more energy and amplify more as they rise up the shoreface before breaking:

PeriodShoal FactorExample
< 6s (windswell)1.0×No amplification
10s~1.2×1.0m → 1.2m
14s~1.4×1.0m → 1.4m
≥ 18s (groundswell)1.6×1.0m → 1.6m
effective_height = offshore_height × Kd × shoal_factor(period, slope)

The shoal factor is now adjusted per beach based on shoreface slope:

Slope classModifierBeaches
Flat×0.85Long Reef, Dee Why
Moderate×1.00Curl Curl, Freshwater
Steep×1.15North Steyne, South Steyne

This is why a 0.8m wave at Dee Why (flat) produces barely a ripple while the same swell at South Steyne (steep) generates a clean breaking wave.

All heights are face height in Australian convention — the vertical trough-to-crest of the breaking wave.

🌊 Multi-Swell Evaluation

The Open-Meteo spectral wave model partitions the wave spectrum into up to three swell components (primary, secondary, tertiary), each with its own height, period, and direction, plus a separate wind-sea (locally generated chop) component. The system now evaluates all available components for each beach individually.

How It Works

For each beach, the system computes the effective height, attack angle, and overall rating for:

  1. Primary swell — the dominant spectral component
  2. Secondary swell — second spectral component (when available)
  3. Tertiary swell — third spectral component (when available)

The component that produces the best precise rating for that beach is used. This means:

  • A south-facing beach like Long Reef might use the primary S swell
  • An east-facing beach like North Steyne might switch to the secondary E swell if it's larger
  • Each beach gets its own optimal swell component, rather than being forced to use the primary

Swell Type Classification

Rather than labelling components as "primary", "secondary", etc., the conditions table classifies each by its period into meaningful surfing categories:

TypePeriodSourceQuality
Groundswell≥ 10sDistant storms, organised long-period energy🟢 Clean, powerful
Windswell< 10sLocal or regional winds, short-period chop🟡 Weak, less organised
Wind SeaN/ALocally generated wind chop at the beach🔴 Bumpy, messy

The classification uses peak period (when available) or mean period as a fallback. The best component of each type is displayed in its own row in the Conditions table, making it easy to see at a glance whether there's proper groundswell running or just windswell.

Each beach card also shows which swell component was active in its tooltip, e.g. "Active: secondary swell (1.2m @ 12s 80°)" when a non-primary component scored best for that beach.

🌊 Breaker Index Quality

Even when a wave reaches the beach, it may not break cleanly. The breaker index γb = Hb / hb (wave height ÷ water depth at the break line) determines how the wave breaks. The depth at the break line is estimated from tide height and beach slope:

γbBreak typeQuality factor
< 0.6Surging / doesn't break0.4
0.6–0.8Spilling / mushy0.7
0.8–1.0Plunging / clean1.0
> 1.0Collapsing / dumpy0.5

The breaker quality factor is multiplied into the overall wave quality alongside wind, attack angle, tide, and embayment. This is why small waves on a flat beach at low tide often score lower than you'd expect from the height alone — they simply don't rear up into a breaking face.

🌊 Tide Height Calculation — 8-Constituent Harmonic Model

The system uses a harmonic tide model with 8 astronomical constituents calibrated for Fort Denison (Sydney Harbour), covering over 95% of tidal variance:

tide(t) = MSL + Σ Ai × sin(2π × t / Ti − φi)
ConstituentAmplitude (m)Period (h)Description
M20.5212.42Principal lunar — largest component
S20.2112.00Principal solar — spring/neap driver
N20.1012.66Lunar elliptic — modulates spring/neap intensity
K20.0611.97Lunisolar — diurnal inequality
K10.0823.93Lunisolar diurnal — daily asymmetry
O10.0525.82Principal lunar diurnal — daily asymmetry
P10.0324.07Principal solar diurnal — daily asymmetry
Q10.0226.87Lunar elliptic diurnal — minor refinement

MSL = 0.9 m (Mean Sea Level relative to Chart Datum). The model uses an epoch-based calculation (since Jan 1, 2000). Phase angles are calibrated for Sydney Harbour.

Tide Trend, Speed & Range

The trend (rising/falling/slack) is determined by the harmonic model directly. Spring vs neap classification uses the tidal range (spring if > 1.2m). Tide speed reflects how fast the water is moving through each 3-hour forecast window — fastest at mid-tide, slack near high/low.

⭐ Surf Quality Rating — Color-Coded Badges

Each beach's rating is displayed as a color-coded pill badge instead of stars — instantly scannable by colour:

ScoreBadgeColour
4.5 – 5.0Epic🟣 Purple
3.5 – 4.0Great🟢 Green
2.5 – 3.0Good🔵 Blue
1.5 – 2.0Fair🟡 Amber
0 – 1.0Poor🟠 Red

The badge replaces the old star system (★½☆) and provides a 1:1 mapping from score to label.

1. Wave Height Score

wave_height_score = min(5, 5 × (effective_height / 1.83)0.85)

Capped at 1.83m (6 ft / overhead) — beyond that, height plateaus and quality factors differentiate.

2. Wave Quality Factor (multiplicative)

wave_quality = wind_quality × attack_factor × tide_factor × embayment_factor × breaker_quality × wind_sea_quality × groundswell_factor

Seven factors multiplied together (not averaged) — one bad factor heavily penalises the score.

3. Groundswell Quality Factor

Rewards organised long-period energy and penalises short-period windswell. The factor is applied per swell component based on its period:

PeriodFactorMeaning
≥ 12s1.30Proper long-period groundswell, clean organised lines
≥ 10s1.15Decent groundswell, good energy
≥ 8s1.00Mixed, some organisation
≥ 6s0.75Short-period windswell, weak/pitchy
< 6s0.50Wind-driven chop

This means a 1.2m wave at 12s scores significantly higher than the same height at 6s, even before considering the effective height calculation (which also rewards longer periods through shoaling amplification).

Final Rating

rating = wave_height_score × (0.5 + wave_quality / 2)

Wave height carries twice the weight of quality. Because quality is a product, a single bad factor (e.g. onshore wind) ruins the score regardless of wave height — just like in real surfing.

🏄 Board Recommendations

Board selection is based on effective face height, with adjustments for wave period and quality. On poor quality days (onshore wind, ugly tide, sloppy conditions), the effective height is adjusted downward so the system recommends more volume.

Base Thresholds (ideal conditions)

Face HeightFeetBoards
< 0.3m< 1 ftLongboard, Log
0.3–0.6m1–2 ftLongboard, Funboard
0.6–1.0m2–3 ftGroveller, Fish, Funboard
1.0–1.5m3–5 ftShortboard, Fish, Mid-Length
1.5–2.0m5–7 ftShortboard, Mid-Length, Step-Up
2.0–2.5m7–8 ftShortboard, Step-Up
> 2.5m> 8 ftStep-Up

Period Adjustments

  • < 8s (windswell): Favour more volume — e.g. Shortboard → Fish/Funboard
  • ≥ 12s (groundswell) + ≥ 1.2m: Favour Step-Up for power

Quality Adjustment

When wave_quality is poor, the effective height is reduced for board selection:

adj_height = effective_height × (0.7 + wave_quality × 0.3)

Perfect conditions (quality = 1.0) → adj = effective height (no change). Terrible conditions (quality = 0.2) → adj = 76% of actual (bump up one volume category).

🌡️ Wetsuit Guide

Wetsuit recommendations are based on real-time sea surface temperature from the IMOS RAMSSA L4 satellite analysis, queried daily:

  • ≥ 22°C: Boardshorts or rash vest
  • 20–21°C: Spring suit (2 mm)
  • 17–19°C: 3/2 steamer
  • 14–16°C: 4/3 steamer
  • < 14°C: 5/4 steamer with booties, gloves, hood

Falls back to monthly climatology if the satellite data is unavailable.

🏞️ Embayment Factor

The embayment factor measures how well a beach's physical layout accommodates the incoming swell. It answers the question: "Is this beach wide enough for this wave, or will it close out?"

The Theory

Surf science defines a key ratio:

W / Lb

Where W is the beach width between headlands (proxied by the total angular window in degrees), and Lb is the wave's sub-wavelength — how much lateral room the wave rotation needs.

  • W / Lb < 2 → The beach is too narrow. Wave energy cannot spread out; the basin fills up faster than it drains → close-out.
  • W / Lb ≥ 6 → Plenty of lateral room. Sandbars fracture into deep channels; water escapes. Clean, triangular A-frame peaks peel left and right → sweet spot.

Our Implementation

We compute the embayment factor from two inputs:

W = left_offset + right_offset (degrees, from beaches.json)
Demand = effective_height × (period² / 100)
Ratio = W / Demand
embayment_factor = map(Ratio → 0.30–1.00)

The period² term is crucial — longer-period groundswell carries exponentially more energy and needs far more lateral room. A 1m wave at 6s and a 1m wave at 16s are completely different beasts in an enclosed bay.

Real-World Behaviour

BeachWindowSmall day (0.7m@8s)Big day (2.5m@14s)
Freshwater57.5°1.00 ✅ fun punchy peaks0.53 ⚠️ close-out risk
Long Reef90°1.00 ✅ plenty of room0.63 ✅ handles it
Curl Curl90°1.00 ✅ plenty of room0.63 ✅ handles it

This matches local knowledge: Freshwater closes out on big southerly groundswell while Long Reef and Curl Curl handle it. On small days, Freshwater's tight profile actually sharpens the waves into fun, punchy peaks.

Display

Rather than a separate column, the embayment factor is shown as a conditional emoji next to the beach name on each card:

  • ≥ 75% — no emoji (beach has enough room) ✅
  • 50–75% — ⚠️ warning (constrained, marginal quality)
  • < 50% — 🚫 danger (close-out risk, seek a wider beach)

This keeps the card layout clean while giving an instant visual read on which beaches are constrained by their layout for today's particular swell size and period.

🤖 LLM-Powered Surf Notes

When the OPENROUTER_API_KEY environment variable is set, the system generates a brief, one-sentence surf note for each beach at each timeframe using an LLM (DeepSeek via OpenRouter).

The LLM receives all the computed numerical data — swell height/period, embayment, attack angle, wind quality, tide factor — and synthesises it into a concise, colloquial note explaining the overall experience and the key factor driving it. Notes are cached using a content-addressed SHA-256 hash of the prompt — when the underlying data hasn't changed, the cached response is reused, avoiding redundant API calls.

Each note appears as a green-tinted bubble below the beach card data. The LLM call is made once for all timeframes in a single API request. A 7-day outlook note is also generated.

📁 Project Architecture

The codebase is organised into focused modules:

  • surf_report.py — Orchestrator: fetches data, computes conditions, renders templates, writes output
  • data/ — Data modules: config.py, marine.py, weather.py, wind.py, tides.py, surf.py
  • templates/ — Jinja2 HTML templates: base.html, timeframe.html, beach_card.html, conditions_grid.html, outlook.html, error.html
  • css/ — CSS source files (style.css, error.css) copied to docs/ at build time
  • imos_sst.py — Real-time SST from IMOS S3
  • llm_report.py — LLM surf commentary
  • datastore.py — SQLite cache layer

HTML is rendered via Jinja2 templates (not f-strings), keeping presentation separate from logic. Never edit files in docs/ directly — edit the source files and run ./venv/bin/python surf_report.py to regenerate.

🤖 Automation & Deployment

  1. 5:45 AM daily — macOS launch agent triggers surf_report.py
  2. Python script fetches live data from Open-Meteo APIs
  3. Queries IMOS S3 bucket for real-time SST
  4. Calculates beach-specific conditions, ratings, and recommendations
  5. Calls OpenRouter API for LLM-powered surf notes (content-addressed cache avoids redundant calls)
  6. Renders HTML via Jinja2 templates (templates/*.html) with color-coded quality badges
  7. Copies CSS source files (css/style.css, css/error.css) to docs/
  8. Writes docs/index.html
  9. Commits changes to the local git repository
  10. Pushes to GitHub, which triggers automatic GitHub Pages rebuild
  11. Live site updates within minutes

⚠️ Known Limitations

  • Wiegel diffraction curves assume idealised headland geometry
  • Single offshore data point for all beaches — no local wind effects
  • No tide graph — tide is shown as text only
  • Board recommendations are static rules, not per-beach calibrated
