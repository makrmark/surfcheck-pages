\U0001f30a Overview

Northern Beaches Surf Check is an automated surf reporting system for Sydney's Northern Beaches. It runs daily at 5:45 AM via a cron job on WSL2, fetches live marine and meteorological data, processes it through a physics-based surf model, generates a detailed HTML report, and deploys it to GitHub Pages.

This page explains how the surf report is generated \u2014 the data sources, calculations, and decisions behind every rating and recommendation. The system evaluates three swell components (primary, secondary, tertiary) from the spectral wave model, classifies them by period into Groundswell and Windswell, and uses the best component for each beach's rating.

\U0001f4e1 Data Sources

IMOS Satellite SST (Real-Time Water Temperature)

Instead of relying on monthly averages, the system fetches real-time sea surface temperature from the Integrated Marine Observing System (IMOS) via their public S3 bucket:

  • Product: RAMSSA L4 \u2014 Regional Australian Multi-Sensor SST Analysis (9 km resolution, gap-free)
  • Bucket: s3://imos-data/IMOS/SRS/SST/ghrsst/L4/RAMSSA/
  • Access: Public bucket, accessed via aws s3 cp --no-sign-request
  • Query: Downloads the latest NetCDF file, extracts SST for the Northern Beaches coastal edge (first ocean pixel offshore)
  • Coverage: Daily 12:00 UTC analyses, typically available by ~17:00 UTC the same day

If the S3 endpoint is unreachable or the NetCDF file cannot be parsed, the system falls back to monthly climatological averages.

Open-Meteo Marine API

Provides hourly wave forecasts for the Sydney coast (-33.78, 151.30) using the ECMWF WAM spectral wave model:

  • Swell Components \u2014 The model partitions the wave spectrum into up to three swell components (primary, secondary, tertiary). Each has its own height, period, and direction. The system evaluates all three and picks the best one per beach.
  • Wave Height \u2014 Significant wave height in metres (Hs)
  • Wave Period \u2014 Peak wave period in seconds
  • Wave Direction \u2014 Direction waves are coming from, in degrees from north
  • Wind Sea \u2014 Locally-generated wind chop, shown separately in the conditions table
  • Wind Speed & Direction \u2014 10m wind speed and direction

Tide Data \u2014 Harmonic Tide Model (8 Constituents)

The system uses a harmonic tide model with 8 constituents (M2, S2, N2, K2, K1, O1, P1, Q1) calibrated for Fort Denison. See the Tide section below for full details. The model covers >95% of tidal variance.

\U0001f30a Effective Wave Height at Each Beach

The system computes effective height \u2014 the face height of the breaking wave at each beach \u2014 by applying two corrections to the offshore swell:

1. Headland Diffraction \u2014 Wiegel's Curves

Each beach has a primary aspect and an L/R window defined in beaches.json. Swell arriving inside the window is fully exposed (Kd = 1.0). Outside the window, energy is reduced by Wiegel diffraction:

Shadow angleKdHeight retained
0\u00b0 (inside window)1.00100%
10\u00b0 past0.7070%
30\u00b0 past0.4040%
60\u00b0 past0.22522.5%
90\u00b0 past0.1515%

Kd is adjusted for wave period: longer-period groundswell (16s) wraps around headlands more effectively, increasing Kd by up to 1.24\u00d7, while short-period windswell (6s) diffracts less, reducing Kd to 0.84\u00d7 via 0.6 + 0.04 \u00d7 T.

2. Shoaling Amplification

Longer-period waves carry more energy and amplify more as they rise up the shoreface before breaking:

PeriodShoal FactorExample
< 6s (windswell)1.0\u00d7No amplification
10s~1.2\u00d71.0m \u2192 1.2m
14s~1.4\u00d71.0m \u2192 1.4m
\u2265 18s (groundswell)1.6\u00d71.0m \u2192 1.6m
effective_height = offshore_height \u00d7 Kd \u00d7 shoal_factor(period, slope)

The shoal factor is adjusted per beach based on shoreface slope:

Slope classModifierBeaches
Flat\u00d70.85Long Reef, Queenscliff, North Steyne, South Steyne
Moderate\u00d71.00Dee Why
Steep\u00d71.15North Curl Curl, South Curl Curl, Freshwater

This is why a 0.8m wave at Long Reef (flat) produces barely a ripple while the same swell at North Curl Curl (steep) generates a clean breaking wave.

All heights are face height in Australian convention \u2014 the vertical trough-to-crest of the breaking wave.

\U0001f30a Multi-Swell Evaluation

The Open-Meteo spectral wave model partitions the wave spectrum into up to three swell components (primary, secondary, tertiary), each with its own height, period, and direction, plus a separate wind-sea (locally generated chop) component. The system evaluates all available components for each beach individually.

How It Works

For each beach, the system computes the effective height, attack angle, and overall rating for:

  1. Primary swell \u2014 the dominant spectral component
  2. Secondary swell \u2014 second spectral component (when available)
  3. Tertiary swell \u2014 third spectral component (when available)

The component that produces the best precise rating for that beach is used. This means:

  • A south-facing beach like Long Reef might use the primary S swell
  • An east-facing beach like North Steyne might switch to the secondary E swell if it's larger
  • Each beach gets its own optimal swell component, rather than being forced to use the primary

Swell Type Classification

Rather than labelling components as "primary", "secondary", etc., the conditions table classifies each by its period into meaningful surfing categories:

TypePeriodSourceQuality
Groundswell\u2265 10sDistant storms, organised long-period energy\U0001f7e2 Clean, powerful
Windswell< 10sLocal or regional winds, short-period chop\U0001f7e1 Weak, less organised
Wind SeaN/ALocally generated wind chop at the beach\U0001f534 Bumpy, messy

The classification uses peak period (when available) or mean period as a fallback. The best component of each type is displayed in its own row in the Conditions table, making it easy to see at a glance whether there's proper groundswell running or just windswell.

Each beach also shows which swell component was active in its tooltip, e.g. "Active: secondary swell (1.2m @ 12s 80\u00b0)" when a non-primary component scored best for that beach.

\U0001f30a Breaker Index Quality

Even when a wave reaches the beach, it may not break cleanly. The breaker index \u03b3b = Hb / hb (wave height \u00f7 water depth at the break line) determines how the wave breaks. The depth at the break line is estimated from tide height and beach slope:

\u03b3bBreak typeQuality factor
< 0.6Surging / doesn't break0.4
0.6\u20130.8Spilling / mushy0.7
0.8\u20131.0Plunging / clean1.0
> 1.0Collapsing / dumpy0.5

The breaker quality factor is multiplied into the overall wave quality alongside wind, attack angle, tide, and embayment. This is why small waves on a flat beach at low tide often score lower than you'd expect from the height alone \u2014 they simply don't rear up into a breaking face.

\U0001f30a Tide Height Calculation \u2014 8-Constituent Harmonic Model

The system uses a harmonic tide model with 8 astronomical constituents calibrated for Fort Denison (Sydney Harbour), covering over 95% of tidal variance:

tide(t) = MSL + \u03a3 Ai \u00d7 sin(2\u03c0 \u00d7 t / Ti \u2212 \u03c6i)
ConstituentAmplitude (m)Period (h)Description
M20.5212.42Principal lunar \u2014 largest component
S20.2112.00Principal solar \u2014 spring/neap driver
N20.1012.66Lunar elliptic \u2014 modulates spring/neap intensity
K20.0611.97Lunisolar \u2014 diurnal inequality
K10.0823.93Lunisolar diurnal \u2014 daily asymmetry
O10.0525.82Principal lunar diurnal \u2014 daily asymmetry
P10.0324.07Principal solar diurnal \u2014 daily asymmetry
Q10.0226.87Lunar elliptic diurnal \u2014 minor refinement

MSL = 0.9 m (Mean Sea Level relative to Chart Datum). The model uses an epoch-based calculation (since Jan 1, 2000). Phase angles are calibrated for Sydney Harbour.

Tide Trend, Speed & Range

The trend (rising/falling/slack) is determined by the harmonic model directly. Spring vs neap classification uses the tidal range (spring if > 1.2m). Tide speed reflects how fast the water is moving through each 3-hour forecast window \u2014 fastest at mid-tide, slack near high/low. Each timeframe includes a small inline SVG tide sparkline showing the 3-hour trend.

\u2b50 Surf Quality Rating \u2014 Color-Coded Badges

Each beach's rating is displayed as a color-coded pill badge instead of stars \u2014 instantly scannable by colour:

ScoreBadgeColour
4.5 \u2013 5.0Epic\U0001f7e3 Purple
3.5 \u2013 4.0Great\U0001f7e2 Green
2.5 \u2013 3.0Good\U0001f535 Blue
1.5 \u2013 2.0Fair\U0001f7e1 Amber
0 \u2013 1.0Poor\U0001f7e0 Red

The badge replaces the old star system (\u2605\u00bd\u2606) and provides a 1:1 mapping from score to label.

1. Wave Height Score

wave_height_score = min(5, 5 \u00d7 (effective_height / 1.83)0.85)

Capped at 1.83m (6 ft / overhead) \u2014 beyond that, height plateaus and quality factors differentiate.

2. Wave Quality Factor (multiplicative)

wave_quality = wind_quality \u00d7 attack_factor \u00d7 tide_factor \u00d7 embayment_factor \u00d7 breaker_quality \u00d7 wind_sea_quality \u00d7 groundswell_factor

Seven factors multiplied together (not averaged) \u2014 one bad factor heavily penalises the score.

3. Groundswell Quality Factor

Rewards organised long-period energy and penalises short-period windswell. The factor is applied per swell component based on its period:

PeriodFactorMeaning
\u2265 12s1.30Proper long-period groundswell, clean organised lines
\u2265 10s1.15Decent groundswell, good energy
\u2265 8s1.00Mixed, some organisation
\u2265 6s0.75Short-period windswell, weak/pitchy
< 6s0.50Wind-driven chop

This means a 1.2m wave at 12s scores significantly higher than the same height at 6s, even before considering the effective height calculation (which also rewards longer periods through shoaling amplification).

Final Rating

rating = wave_height_score \u00d7 (0.5 + wave_quality / 2)

Wave height carries twice the weight of quality. Because quality is a product, a single bad factor (e.g. onshore wind) ruins the score regardless of wave height \u2014 just like in real surfing.

\U0001f3c4 Board Recommendations

Board selection is based on effective face height, with adjustments for wave period and quality. On poor quality days (onshore wind, ugly tide, sloppy conditions), the effective height is adjusted downward so the system recommends more volume.

Base Thresholds (ideal conditions)

Face HeightFeetBoards
< 0.3m< 1 ftLongboard, Log
0.3\u20130.6m1\u20132 ftLongboard, Funboard
0.6\u20131.0m2\u20133 ftGroveller, Fish, Funboard
1.0\u20131.5m3\u20135 ftShortboard, Fish, Mid-Length
1.5\u20132.0m5\u20137 ftShortboard, Mid-Length, Step-Up
2.0\u20132.5m7\u20138 ftShortboard, Step-Up
> 2.5m> 8 ftStep-Up

Period Adjustments

  • < 8s (windswell): Favour more volume \u2014 e.g. Shortboard \u2192 Fish/Funboard
  • \u2265 12s (groundswell) + \u2265 1.2m: Favour Step-Up for power

Quality Adjustment

When wave_quality is poor, the effective height is reduced for board selection:

adj_height = effective_height \u00d7 (0.7 + wave_quality \u00d7 0.3)

Perfect conditions (quality = 1.0) \u2192 adj = effective height (no change). Terrible conditions (quality = 0.2) \u2192 adj = 76% of actual (bump up one volume category).

\U0001f321\ufe0f Wetsuit Guide

Wetsuit recommendations are based on a combined feels-like index that blends real-time SST (IMOS RAMSSA L4) with windchill-adjusted air temperature. Each timeframe gets its own recommendation based on its own windchill conditions.

  • \u2265 22\u00b0C: Boardshorts or rash vest
  • 20\u201321\u00b0C: Spring suit (2 mm)
  • 17\u201319\u00b0C: 3/2 steamer
  • 14\u201316\u00b0C: 4/3 steamer
  • < 14\u00b0C: 5/4 steamer with booties, gloves, hood

Windchill (NWS formula) is applied when air temp \u2264 10\u00b0C and wind \u2265 4.8 km/h. Combined temp = 0.65\u00d7SST + 0.35\u00d7windchill_air. Falls back to monthly climatology if satellite data is unavailable.

\U0001f3de\ufe0f Embayment Factor

The embayment factor measures how well a beach's physical layout accommodates the incoming swell. It answers the question: "Is this beach wide enough for this wave, or will it close out?"

The Theory

Surf science defines a key ratio:

W / Lb

Where W is the beach width between headlands (proxied by the bay_width attribute: wide \u2192 90\u00b0, narrow \u2192 57.5\u00b0), and Lb is the wave's energy demand \u2014 how much lateral room the wave rotation needs.

  • W / Lb < 2 \u2192 The beach is too narrow. Wave energy cannot spread out; the basin fills up faster than it drains \u2192 close-out.
  • W / Lb \u2265 6 \u2192 Plenty of lateral room. Sandbars fracture into deep channels; water escapes. Clean, triangular A-frame peaks peel left and right \u2192 sweet spot.

Our Implementation

We compute the embayment factor from two inputs:

W = bay_width (degrees, from beaches.json)
Demand = effective_height \u00d7 (period\u00b2 / 100)
Ratio = W / Demand
embayment_factor = map(Ratio \u2192 0.30\u20131.00)

The period\u00b2 term is crucial \u2014 longer-period groundswell carries exponentially more energy and needs far more lateral room. A 1m wave at 6s and a 1m wave at 16s are completely different beasts in an enclosed bay.

Real-World Behaviour

BeachBaySmall day (0.7m@8s)Big day (2.5m@14s)
FreshwaterNarrow1.00 \u2705 fun punchy peaks0.53 \u26a0\ufe0f close-out risk
Long ReefWide1.00 \u2705 plenty of room0.63 \u2705 handles it
South Curl CurlWide1.00 \u2705 plenty of room0.63 \u2705 handles it

This matches local knowledge: Freshwater closes out on big southerly groundswell while Long Reef and Curl Curl handle it. On small days, Freshwater's tight profile actually sharpens the waves into fun, punchy peaks.

Display

Rather than a separate column, the embayment factor is shown as a conditional emoji next to the beach name on each card:

  • \u2265 75% \u2014 no emoji (beach has enough room)
  • 50\u201375% \u2014 \u26a0\ufe0f warning (constrained, marginal quality)
  • < 50% \u2014 \U0001f6ab danger (close-out risk, seek a wider beach)

This keeps the card layout clean while giving an instant visual read on which beaches are constrained by their layout for today's particular swell size and period.

\U0001f916 LLM-Powered Surf Notes

When the OPENROUTER_API_KEY environment variable is set, the system generates surf commentary using an LLM (DeepSeek via OpenRouter) at three levels:

  • Daily assessment \u2014 A paragraph summarising the day's conditions across all beaches, displayed at the top of each day
  • Per-beach notes \u2014 A one-sentence qualitative note for each beach, displayed in the beach detail card
  • 7-day outlook \u2014 A forecast summary for the week ahead, shown in the outlook section

The LLM receives all the computed numerical data \u2014 swell height/period, embayment, attack angle, wind quality, tide factor \u2014 and synthesises it into concise, colloquial notes explaining the overall experience and the key factor driving it. Notes are cached using a content-addressed SHA-256 hash of the prompt \u2014 when the underlying data hasn't changed, the cached response is reused, avoiding redundant API calls. Every prompt and response is logged to ~/.surforecast/llm_training_log.jsonl for future fine-tuning.

\U0001f4c1 Project Architecture

The codebase is organised into focused modules:

  • surf_report.py \u2014 Orchestrator: fetches data, computes conditions, renders templates, writes output. Also builds the beach\u00d7timeframe matrices and conditions matrices for the daily overview.
  • data/ \u2014 Data modules: config.py (config, logging, beach loading), marine.py, weather.py, wind.py, tides.py (harmonic tide model), surf.py (ratings, exposure, diffraction, embayment, board recs), charts.py (SVG chart generation)
  • templates/ \u2014 Jinja2 HTML templates: base.html (full page layout with day nav), daily_overview.html (per-day beach matrix, conditions matrix, beach cards), error.html (fallback error page)
  • css/ \u2014 CSS source files (style.css, error.css) copied to docs/ at build time
  • imos_sst.py \u2014 Real-time SST from IMOS S3 RAMSSA L4
  • llm_report.py \u2014 LLM surf commentary (daily assessment, per-beach notes, outlook)
  • datastore.py \u2014 SQLite-backed API response cache with TTL expiry and stale fallback
  • clear_cache.py \u2014 CLI cache invalidation script
  • beaches.json \u2014 Beach configuration (aspect, L/R offsets, slope, bay_width)
  • deploy.sh \u2014 Deployment script that runs surf_report.py, commits, and pushes to GitHub Pages

HTML is rendered via Jinja2 templates (not f-strings), keeping presentation separate from logic. Never edit files in docs/ directly \u2014 edit the source files and run ./venv/bin/python surf_report.py to regenerate.

\U0001f916 Automation & Deployment

  1. 5:45 AM daily \u2014 WSL2 cron job triggers deploy.sh
  2. deploy.sh sources environment variables, activates the Python venv, and runs surf_report.py
  3. Python script fetches live data from Open-Meteo APIs (marine, weather, wind)
  4. Queries IMOS S3 bucket for real-time SST
  5. Calculates beach-specific conditions, ratings, and recommendations for 8 timeframes \u00d7 8 beaches
  6. Calls OpenRouter API for LLM-powered surf commentary (daily assessment, per-beach notes, 7-day outlook) \u2014 content-addressed cache avoids redundant calls
  7. Renders HTML via Jinja2 templates (templates/base.html, templates/daily_overview.html) with color-coded quality badges
  8. Copies CSS source files (css/style.css, css/error.css) to docs/
  9. Writes docs/index.html with inline CSS
  10. Commits changes to the local git repository
  11. Pushes to GitHub, which triggers automatic GitHub Pages rebuild
  12. Live site updates within minutes

\u26a0\ufe0f Known Limitations

  • Wiegel diffraction curves assume idealised headland geometry \u2014 real headlands have irregular shapes, cliffs, and varying distances to the beach
  • Single offshore data point for all beaches \u2014 no local wind effects or terrain channelling
  • No tide graph \u2014 tide is shown as an inline SVG sparkline, but no full daily curve
  • Board recommendations are static rules, not per-beach calibrated
  • SST is a single coastal value; real-world variation of 1\u20132\u00b0C between beaches is not captured
  • IMOS SST pipeline requires aws s3 cp --no-sign-request and numpy on the host
